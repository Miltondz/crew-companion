- **§4** — Tool decorator: rewritten to wrap the original Python function BEFORE LangChain's `@tool`, preserving signature + `InjectedState`. Returns `Command` for state-mutating tools or `ToolMessage` content for read-only.
- **§5** — `PolicyDecision` types pulled to module scope; rule order corrected; approval-token validation added.

## Decision

Introduce a governance layer between tool declaration and tool execution. Every tool declares capabilities + risk level at import time via `@guarded_tool`. At call time, `PolicyEngine.evaluate()` returns Allow / Deny / PendingApproval before the tool body runs. Denials and pending-approval states are returned as structured error envelopes — the tool body is never entered. Audit rows are written (to console / stub in 3.3; real DB in 3.4). The frontend receives new envelope intents (`request_approval`, `tool_denied`) and routes them to an `ApprovalGate` component.

## Rationale

- Tools already return `Command(update=...)` — we can intercept cleanly in the decorator without modifying LangGraph internals.
- `requiredCapabilities: list[str]` is already on `SurfaceEnvelope` — extending intent literals to `request_approval` / `tool_denied` is minimal delta.
- Stub everything that needs DB (capability storage, real audit rows, rate limiting) — 3.3 wires the shape, 3.4 fills it.
- Frontend-only tools must also declare capabilities — use a `useGuardedFrontendTool` wrapper that validates against BFF-vended role context at hook invocation time (no round-trip needed; context is already in React state).

Risk semantics:
- **LOW** — reads, UI hints, mascot mood. No side-effect concern.

- **HIGH** — delete, invite, db.query. Requires explicit role check; some require approval.
- **CRITICAL** — db.mutate, fs.write, shell.execute. Blocked entirely in 3.3; future: require approval + sandbox.

## Part 4 — Tool Decorator (`@guarded_tool`) — REWRITTEN (Opus)

### 4.1 Skeleton

**Critical fix:** Sonnet's original wrapper double-wrapped `@lc_tool`
around a `(*args, **kwargs)` shim, which **strips the original function's
signature** — LangChain's `@tool` introspects parameter names, types, and
`InjectedState` annotations to build the tool's JSON schema for the LLM.
Wrapping `(*args, **kwargs)` produces an empty schema; the LLM cannot
call the tool with structured arguments. The rewrite below interposes the
guard *inside* the LangChain-decorated function via a closure, preserving
the original signature.

```python
# apps/agent/src/runtime/tool_decorator.py

from __future__ import annotations

import inspect
from dataclasses import dataclass
from functools import wraps
from typing import Callable, List, Optional

from langchain_core.runnables import RunnableConfig
from langchain_core.tools import tool as lc_tool

from .capabilities import Capability, RiskLevel
from .policy import PolicyEngine, ExecutionContext, PolicyDecision, Deny, PendingApproval, Allow
from .audit import AuditLogger

# Global registry — populated at import time
_TOOL_REGISTRY: dict[str, "ToolMeta"] = {}


@dataclass(frozen=True)
class ToolMeta:
    """Immutable. Risk level is a code decision, not a runtime knob."""
    tool_id: str
    capabilities: tuple[Capability, ...]
    risk_level: RiskLevel
    requires_approval: bool
    audit: bool
    sandbox: Optional[str]


def guarded_tool(
    *,
    capabilities: list[Capability],
    risk_level: RiskLevel,
    requires_approval: bool = False,
    audit: bool = True,
    sandbox: Optional[str] = None,
):
    """
    Decorator that:
      1. Records ToolMeta in _TOOL_REGISTRY at import time
      2. Wraps the ORIGINAL function (signature preserved) so LangChain's
         @tool can still introspect parameter names + InjectedState
      3. At call time, invokes PolicyEngine BEFORE the body
      4. On Deny → returns a ToolMessage-compatible content string AND
         writes audit row. The agent observes the denial in its next turn
         and surfaces it to the user via natural language.
      5. On PendingApproval → raises `langgraph.types.interrupt(...)` —
         the graph suspends. See §13 for resume mechanics.
      6. On Allow → calls the original function, audits success/error.

    The decorator preserves the original function's signature including
    `Annotated[dict, InjectedState]` and `config: RunnableConfig` so the
    LLM gets the right tool schema and LangGraph still injects state/config.
    """
    def decorator(fn: Callable) -> Callable:
        sig = inspect.signature(fn)

        meta = ToolMeta(
            tool_id=fn.__name__,
            capabilities=tuple(capabilities),
            risk_level=risk_level,
            requires_approval=requires_approval,
            audit=audit,
            sandbox=sandbox,
        )
        _TOOL_REGISTRY[fn.__name__] = meta

        # Build the wrapper with the SAME signature as fn, so @lc_tool's
        # schema-builder sees the original params (incl. InjectedState).
        @wraps(fn)
        def wrapper(*args, **kwargs):
            # 1. Resolve state and config without removing them from kwargs
            #    (LangGraph injects both as named kwargs).
            state = kwargs.get("state") or {}
            config: RunnableConfig = kwargs.get("config") or {}
            configurable = (config.get("configurable") if isinstance(config, dict) else {}) or {}

            # 2. Build ExecutionContext from config (authoritative) with
            #    state as fallback for fields the BFF wrote there.
            context = ExecutionContext(
                agent_id=configurable.get("agent_id", "orchestrator"),
                workspace_id=configurable.get("workspace_id")
                             or state.get("workspaceId", ""),
                urgency_phase=state.get("urgencyPhase", "normal"),
                role=configurable.get("actor_role")
                     or state.get("actorRole", "member"),
                approval_token=configurable.get("approval_token"),
                approved_tool_call_id=configurable.get("approved_tool_call_id"),
            )

            decision = PolicyEngine().evaluate(meta, context)

            if isinstance(decision, Deny):
                if audit:
                    AuditLogger.log(meta, context, decision="denied",
                                    reason=decision.reason)
                # Return a string — LangGraph converts the tool's return
                # value into a ToolMessage; the model sees the denial and
                # can apologise or pivot. Do NOT raise; that would crash
                # the run.
                return (
                    f"[denied] {meta.tool_id}: {decision.reason}. "
                    f"Inform the user that this action is not permitted "
                    f"in the current context."
                )

            if isinstance(decision, PendingApproval):
                if audit:
                    AuditLogger.log(meta, context, decision="pending",
                                    reason=decision.reason)
                # Defer to LangGraph's native interrupt mechanism. See §13.
                # The BFF resumes the thread with Command(resume={...})
                # containing the approval_token, which lands back here on
                # the second pass via config['configurable']['approval_token'].
                from langgraph.types import interrupt
                approval_payload = interrupt({
                    "kind": "request_approval",
                    "tool": meta.tool_id,
                    "capabilities": [c.value for c in capabilities],
                    "risk_level": risk_level.value,
                    "reason": decision.reason,
                    "args_preview": _safe_args_preview(args, kwargs, sig),
                })
                # If we reach this line, the BFF resumed with payload.
                # Re-inject token into config and re-evaluate; if approved,
                # falls through to Allow on the recursive call.
                if not approval_payload or approval_payload.get("decision") != "approved":
                    if audit:
                        AuditLogger.log(meta, context, decision="rejected",
                                        reason="user_rejected")
                    return f"[rejected] {meta.tool_id}: user did not approve."

                # Approved — re-bind config and recurse exactly once.
                kwargs["config"] = _merge_config(config, {
                    "approval_token": approval_payload["approval_token"],
                    "approved_tool_call_id": approval_payload.get("tool_call_id"),
                })
                return wrapper(*args, **kwargs)

            # Allow path
            try:
                result = fn(*args, **kwargs)
            except Exception as e:
                if audit:
                    AuditLogger.log(meta, context, decision="allowed",
                                    outcome="error", outcome_error=str(e))
                raise

            if audit:
                AuditLogger.log(meta, context, decision="allowed",
                                outcome="success")
            return result

        # Critical: copy the signature so @lc_tool sees fn's params.
        wrapper.__signature__ = sig  # type: ignore[attr-defined]

        return lc_tool(wrapper)

    return decorator


def _safe_args_preview(args, kwargs, sig) -> dict:
    """Strip injected state/config; return user-visible params only."""
    bound = sig.bind_partial(*args, **kwargs)
    preview = {}
    for name, value in bound.arguments.items():
        if name in ("state", "config"):
            continue
        preview[name] = value if isinstance(value, (str, int, float, bool)) else str(value)[:200]
    return preview


def _merge_config(base: RunnableConfig, extra: dict) -> RunnableConfig:
    """Shallow-merge into config.configurable."""
    new = dict(base) if isinstance(base, dict) else {}
    configurable = dict(new.get("configurable", {}))
    configurable.update(extra)
    new["configurable"] = configurable
    return new
```

**Key changes from Sonnet's design:**
- `wrapper` keeps `fn`'s signature via `wrapper.__signature__ = sig`. LangChain `@tool` reads this and builds the correct JSON schema.
- `ExecutionContext` sources `agent_id`, `workspace_id`, `role`, `approval_token` from `config['configurable']` (the canonical place RunnableConfig threads per-invocation data through LangGraph), not from state. State fallbacks remain for backward compat.
- Deny returns a **string** (becomes a ToolMessage), not a dict — LangGraph's tool node turns whatever the tool returns into a ToolMessage; a structured-error dict was the right impulse but breaks the contract for tools that should return `Command`. The string form is uniform and the LLM can act on it.
- PendingApproval invokes `langgraph.types.interrupt()` — the actual mechanism LangGraph provides for human-in-the-loop. See §13.
- `ToolMeta` is `frozen=True` and `capabilities` is `tuple` (immutable) — invariant 11 was claimed but not enforced in Sonnet's code.

## Part 5 — PolicyEngine — REWRITTEN (Opus)

### 5.1 Data types

**Critical fix:** Sonnet defined `PolicyDecision.Allow/Deny/PendingApproval`
as nested classes inside `class PolicyDecision`, then declared
`Result = Union[Allow, Deny, PendingApproval]` *inside the same class body*.
This is malformed Python — class-body `Union[...]` is evaluated against
names already bound to the class namespace and Result becomes a class
attribute, not a usable type alias. Pulled to module scope.

```python
# apps/agent/src/runtime/policy.py

from __future__ import annotations

from dataclasses import dataclass
from typing import Optional, Union

from .capabilities import Capability, RiskLevel


@dataclass(frozen=True)
class ExecutionContext:
    agent_id: str                         # "orchestrator" | future
    workspace_id: str
    urgency_phase: str                    # "normal" | "focus" | "urgent" | "panic" | "expired"
    role: str                             # "leader" | "member" | "viewer"
    approval_token: Optional[str] = None
    approved_tool_call_id: Optional[str] = None  # bound to a specific call


# Module-level discriminated union — usable as PolicyDecision type alias.
@dataclass(frozen=True)
class Allow:
    pass


@dataclass(frozen=True)
class Deny:
    reason: str


@dataclass(frozen=True)
class PendingApproval:
    reason: str


PolicyDecision = Union[Allow, Deny, PendingApproval]
```

### 5.2 PolicyEngine class skeleton — rules reordered (Opus)

**Rule order matters.** Cheap pre-flight checks (namespace, role, phase,
rate-limit) run BEFORE the expensive approval round-trip. Sonnet's
original order would have hit rate-limit AFTER user already clicked
Approve — wasteful UX. Approval is the last gate.

```python
class PolicyEngine:
    # Stateless in 3.3. In 3.4 will accept a DB session for
    # workspace_members.capabilities lookup.

    # Allowed top-level namespaces. Any capability outside this set is
    # denied at the namespace level — this is how future caps (db.*, fs.*,
    # shell.*) stay blocked WITHOUT being declared as enum members.
    ALLOWED_NAMESPACES: frozenset[str] = frozenset({
        "state", "tasks", "blockers", "milestones",
        "members", "docs", "ui",
    })

    def evaluate(self, tool: "ToolMeta", context: ExecutionContext) -> PolicyDecision:
        # Rule 1: namespace whitelist — any cap outside the allowed
        # top-level namespaces is denied. Catches db.*, fs.*, shell.*
        # without declaring enum values for them.
        bad = [c for c in tool.capabilities if c.value.split(".", 1)[0] not in self.ALLOWED_NAMESPACES]
        if bad:
            return Deny(f"namespace_blocked:{bad[0].value}")

        # Rule 2: role must grant every required capability
        if not self._role_has_all(context.role, tool.capabilities):
            return Deny("missing_capability")

        # Rule 3: phase-based restriction
        phase_block = self._phase_block_reason(tool, context.urgency_phase)
        if phase_block:
            return Deny(phase_block)

        # Rule 4: rate limit (stub — always False in 3.3, real in 4.7)
        if self._rate_limit_exceeded(context.workspace_id, tool.tool_id):
            return Deny("rate_limit_exceeded")

        # Rule 5: approval gate — LAST so cheap rules deny first
        if tool.requires_approval:
            token_ok = self._approval_token_valid(
                context.approval_token,
                expected_workspace=context.workspace_id,
                expected_tool=tool.tool_id,
            )
            if not token_ok:
                return PendingApproval("user_approval_required")

        return Allow()

    def _role_has_all(self, role: str, required: tuple[Capability, ...]) -> bool:
        """
        Stub: role-based grants. In 3.4 this queries
        workspace_members.capabilities (per-member overrides on top of
        role defaults).

        Source of truth for the grant table lives in a JSON file that
        BOTH this module AND the frontend's `useCrewContext` consume —
        see §9.4. Do NOT duplicate the table.
        """
        from .role_grants import role_grants_for
        granted = role_grants_for(role)
        return all(c in granted for c in required)

    # Keep the old name as a thin alias for transition — delete in 3.4.
    def _has_capabilities(self, agent_id, role, required):  # noqa: ARG002
        return self._role_has_all(role, tuple(required))

    # The full grant table — kept inline here ONLY as a docstring artifact;
    # the live table lives in role_grants.py imported above. DO NOT
    # maintain two copies.
    _GRANT_TABLE_DOCS = """
    ROLE_GRANTS canonical table (see role_grants.py for live values):
            "leader": {
                Capability.STATE_READ, Capability.STATE_WRITE,
                Capability.TASKS_READ, Capability.TASKS_WRITE, Capability.TASKS_DELETE,
                Capability.BLOCKERS_READ, Capability.BLOCKERS_WRITE, Capability.BLOCKERS_RESOLVE,
                Capability.MILESTONES_READ, Capability.MILESTONES_WRITE,
                Capability.MEMBERS_READ, Capability.MEMBERS_INVITE,
                Capability.DOCS_READ, Capability.DOCS_WRITE,
                Capability.UI_RENDER_SURFACE, Capability.UI_SET_MASCOT, Capability.UI_HIGHLIGHT,
            },
            "member": {
                Capability.STATE_READ, Capability.STATE_WRITE,
                Capability.TASKS_READ, Capability.TASKS_WRITE,
                Capability.BLOCKERS_READ, Capability.BLOCKERS_WRITE, Capability.BLOCKERS_RESOLVE,
                Capability.MILESTONES_READ,
                Capability.MEMBERS_READ,
                Capability.DOCS_READ, Capability.DOCS_WRITE,
                Capability.UI_RENDER_SURFACE, Capability.UI_SET_MASCOT, Capability.UI_HIGHLIGHT,
            },
            "viewer": {
                Capability.STATE_READ,
                Capability.TASKS_READ,
                Capability.BLOCKERS_READ,
                Capability.MILESTONES_READ,
                Capability.MEMBERS_READ,
                Capability.DOCS_READ,
            },
        }
    """  # end _GRANT_TABLE_DOCS

    def _phase_block_reason(self, tool: "ToolMeta", phase: str) -> Optional[str]:
        """Returns a deny reason string, or None if not phase-blocked."""
        if phase == "panic" and tool.risk_level in (RiskLevel.HIGH, RiskLevel.CRITICAL):
            return f"blocked_in_phase:panic"
        if phase == "expired" and tool.risk_level in (
            RiskLevel.MEDIUM, RiskLevel.HIGH, RiskLevel.CRITICAL
        ):
            return f"blocked_in_phase:expired"
        return None

    def _rate_limit_exceeded(self, workspace_id: str, tool_id: str) -> bool:
        """Stub. Real implementation in 4.7."""
        return False

    def _approval_token_valid(
        self,
        token: Optional[str],
        *,
        expected_workspace: str,
        expected_tool: str,
    ) -> bool:
        """
        In 3.3: a token is valid iff it is non-empty AND the BFF stored it
        in an in-memory `approval_store` keyed by token, with matching
        workspace and tool, and not yet consumed (single-use).

        See `apps/agent/src/runtime/approval_store.py` (new in 3.3).
        Tokens are unsigned in 3.3; cryptographic signing lands in 4.1.

        On consumption, the entry is removed — replay returns False.
        """
        if not token:
            return False
        from .approval_store import consume_token
        record = consume_token(token)
        if not record:
            return False
        if record["workspace_id"] != expected_workspace:
            return False
        if record["tool_id"] != expected_tool:
            return False
        if record["expires_at"] < _now_ms():
            return False
        return True
```

### 6.2 Stub logger (`apps/agent/src/runtime/audit.py`) — REWRITTEN (Opus)

**Critical fix:** Sonnet's stub wrote `outcome=""` whenever outcome was
not specified. The DDL has
`outcome text CHECK (outcome IN ('success', 'error', NULL))` — empty
string fails the CHECK and the 3.4 swap would crash on the first audit
write. Default must be `None`, and JSON serializer must respect it.

Also: Sonnet's signature `log(tool_meta, context, decision, reason, outcome)`
silently coerces the call site `log(meta, context, "allowed", outcome="success")`
to `reason="success"`. Use keyword-only after the first two positional args.

```python
# apps/agent/src/runtime/audit.py
# 3.3: log to stdout. 3.4: writes to audit_log table (call signature
# unchanged — invariant 12).

from __future__ import annotations

import json
import logging
from datetime import datetime, timezone
from typing import Literal, Optional

logger = logging.getLogger("audit")

Decision = Literal["allowed", "denied", "pending", "approved", "rejected"]
Outcome = Literal["success", "error"]


class AuditLogger:
    @staticmethod
    def log(
        tool_meta,
        context,
        *,
        decision: Decision,
        reason: Optional[str] = None,
        outcome: Optional[Outcome] = None,
        outcome_error: Optional[str] = None,
    ) -> None:
        """Append one audit record. Never raises; swallows all errors."""
        try:
            record = {
                "ts": datetime.now(timezone.utc).isoformat(),
                "workspace_id": context.workspace_id,
                "actor_type": f"agent:{context.agent_id}",
                "actor_id": context.agent_id,
                "tool_id": tool_meta.tool_id,
                "capabilities": [c.value for c in tool_meta.capabilities],
                "risk_level": tool_meta.risk_level.value,
                "decision": decision,
                "decision_reason": reason,        # nullable
                "outcome": outcome,               # nullable — None not ""
                "outcome_error": outcome_error,   # nullable
            }
            # `default=str` swallows datetime/Enum oddities without raising.
            logger.info("AUDIT %s", json.dumps(record, default=str))
            # TODO 3.4: replace stdout with
            #   await db.execute(INSERT INTO audit_log ... VALUES ...)
            # Call signature must NOT change.
        except Exception:
            # Invariant 6: audit failures never block tool execution.
            logger.exception("audit_log_failed")
```

The 3.4 DB writer will keep the same signature (`tool_meta`, `context`,
plus keyword-only fields) and add an optional `db_session` param resolved

### 7.3 Frontend routing — REWRITTEN (Opus)

**Sonnet's recommendation rejected.** Routing `request_approval` and
`tool_denied` through `SurfaceHost` violates the Surface Registry's
purity invariant (from 3.1 §7 invariant 1: "A surface MUST NOT be reachable
by any code path that doesn't go through its manifest").

Surfaces are **visual outputs resolved via Zod-validated manifests**.
Approval is **flow control** — it has no manifest, no `preferredZone`,
no `priority`, no lifecycle hints. Forcing it through the manifest
pipeline either bypasses validation (defeats the registry's purpose) or
requires fake manifests for non-surface concepts (defeats the abstraction).

**Sibling channel approach:**

`apps/frontend/src/runtime/flow-channel/RuntimeFlowChannel.tsx` —
a new component, peer to `SurfaceHost`, that handles non-surface envelopes:

```typescript
// apps/frontend/src/runtime/flow-channel/RuntimeFlowChannel.tsx
'use client'

import type { SurfaceEnvelope } from '@/runtime/surface-registry/types'
import { ApprovalGate } from '@/components/runtime/ApprovalGate'
import { ToolDeniedNotice } from '@/components/runtime/ToolDeniedNotice'

interface RuntimeFlowChannelProps {
  envelope: SurfaceEnvelope
}

export function RuntimeFlowChannel({ envelope }: RuntimeFlowChannelProps) {
  switch (envelope.intent) {
    case 'request_approval':
      return <ApprovalGate pendingAction={envelope.payload as any} />
    case 'tool_denied':
      return <ToolDeniedNotice denial={envelope.payload as any} />
    default:
      return null
  }
}
```

At the page level (and in the CopilotKit `renderSurface` tool), the
caller branches on intent ONCE — `render_surface` goes to `SurfaceHost`,
everything else goes to `RuntimeFlowChannel`:

```typescript
// inside the renderSurface frontend tool's `render`:
const env = ...   // adapted envelope
return env.intent === 'render_surface'
  ? <SurfaceHost envelope={env} context={runtimeContext} />
  : <RuntimeFlowChannel envelope={env} />
```

`ToolDeniedNotice` remains as Sonnet specified — a simple dismissible
inline notice.

**Why this matters:** in 3.2 when the Layout Engine starts negotiating
zones, surfaces have a `preferredZone` from their manifest. Approval
prompts don't have a zone — they appear inline at the agent boundary
(in the chat stream, near the tool call that triggered them). Routing
them through SurfaceHost would force them into a zone they don't belong
in, or force a special-case in the Layout Engine. The sibling channel
keeps the two concerns separable.

### 8.4 Approval flow — REWRITTEN (Opus)

Sonnet's flow says "BFF forwards to agent as HTTP callback / agent resumes
tool with _approval_token in context" without specifying how. **This is
the hardest part of the entire design and Sonnet did not solve it.**

There is no built-in "agent resume" in LangChain `create_react_agent` —
when a tool returns, the agent's run continues normally with the tool
return value as a ToolMessage. There is no implicit suspension. To pause
a run waiting for human input and resume it later, LangGraph requires:

1. A **checkpointer** wired on graph compile (memory or postgres).
2. The tool calls **`langgraph.types.interrupt(payload)`** to suspend.
3. The graph **persists checkpoint state to the checkpointer**.
4. The frontend sees the interrupt and displays the approval UI.
5. The BFF later invokes the graph with **`Command(resume={...})`** on
   the same `thread_id`, supplying the approval payload.
6. The graph resumes from the interrupt point; `interrupt()` returns
   the payload, and execution continues.

**See §13 for the protocol specification.** Sonnet's "HTTP callback" is
not enough — it would re-invoke the graph from the start of the user's
turn, losing tool-call context.

Corrected flow:

```
1. Agent calls a guarded tool (e.g. members.invite).
2. @guarded_tool wrapper builds ExecutionContext, calls PolicyEngine.
3. PolicyEngine returns PendingApproval.
4. AuditLogger.log(decision="pending")
5. Wrapper calls `interrupt({ kind: "request_approval", tool, ... })`
   — LangGraph suspends the graph, persists checkpoint.
6. The interrupt surfaces in the agent's stream as a special event.
7. CopilotKit / @ag-ui adapter forwards the interrupt to the frontend.
8. Frontend's `RuntimeFlowChannel` (NOT SurfaceHost) renders
   <ApprovalGate>.
9. User clicks Approve.
10. POST /api/approvals/:threadId/decide  { decision: "approved",
    approval_token, tool_call_id }
11. BFF validates the request, calls the agent's graph with
    Command(resume={ decision: "approved", approval_token, tool_call_id })
    on the same thread_id.
12. Graph resumes at the interrupt site; interrupt() returns the resume
    payload to the wrapper.
13. Wrapper re-injects approval_token into config['configurable'] and
    recurses ONCE — second pass through PolicyEngine sees the valid
    token, returns Allow.
14. Tool body executes, audit row written (decision="allowed",
    after_approval=true).
15. Agent emits render_surface envelope as usual.

User clicks Reject:
- BFF resumes with Command(resume={ decision: "rejected" }).
- Wrapper sees `decision != "approved"`, audits "rejected", returns
  string content to the model.
- Model surfaces denial in natural language; user sees ToolDeniedNotice
  via emit_envelope from the chat node if the model chooses to render.
```

**BFF route (3.3 stub, real in 3.4):**

```
POST /api/approvals/:threadId/decide
Body: {
  decision: "approved" | "rejected",
  approval_token: string,
  tool_call_id: string
}
```

Note **single endpoint, not separate approve/reject**: keeps the
resume-payload shape uniform and lets the BFF route both decisions
through one `Command(resume=...)` invocation.

**Checkpointer requirement:** `runtime.py` must compile graphs with a
checkpointer or interrupts cannot suspend. In 3.3:

```python
# apps/agent/src/runtime.py — minimal addition for 3.3

return create_agent(
    model=llm,
    tools=tools,
    system_prompt=system_prompt,
    middleware=middleware,
    checkpointer=checkpointer,   # <-- REQUIRED for interrupt() to work
)
```

The BFF must pass a stable `thread_id` per workspace (already required
for CopilotKit threading; verify).

### 9.1 Decision: `useGuardedFrontendTool` wrapper — CLARIFIED (Opus)

**Sonnet was right** to keep this client-side. The decision stands.
But the "audit parity" goal is dropped — see §9.2.

**Rationale (unchanged):**
- Frontend tools execute synchronously as CopilotKit state mutations.
- A BFF round-trip would break the synchronous contract and add latency.
- Role context is already in React state.
- Frontend CANNOT enforce hard security; this is a UX guard only.

**What changes (Opus):** Sonnet's hook posts to `/api/audit/frontend-tool`
after executing. That POST is fire-and-forget — UI mutation has already
happened. Backend cannot veto. The audit record is unreliable (network
failure swallows the row). And worst: it creates the **illusion of audit
parity** which engineers may rely on. Drop the POST. Audit parity for
frontend tools is a 4.x concern — when frontend tools route through the
BFF (already on the roadmap for richer state sync), audit happens at the
BFF, server-side, atomically.

### 9.2 Hook skeleton

```typescript
// apps/frontend/src/runtime/capability/useGuardedFrontendTool.ts

import { useCallback } from "react";
import { useCrewContext } from "@/lib/crew/CrewContext";
import { CapabilityId } from "./capabilities";
import { RiskLevel } from "./riskLevel";

interface GuardedToolOptions {
  toolId: string;
  capabilities: CapabilityId[];
  riskLevel: RiskLevel;
  requiresApproval?: boolean;
}

export function useGuardedFrontendTool<TArgs>(
  options: GuardedToolOptions,
  fn: (args: TArgs) => void,
) {
  const { role, grantedCapabilities } = useCrewContext();

  return useCallback(
    (args: TArgs) => {
      // 1. Check role has all required capabilities
      const missing = options.capabilities.filter(
        (c) => !grantedCapabilities.includes(c),
      );
      if (missing.length > 0) {
        console.warn(`[capability] ${options.toolId} denied — missing: ${missing.join(", ")}`);
        return; // Silent deny — UI should not expose ungrantable actions
      }

      // 2. requiresApproval — frontend tools should never need this in current scope
      // If a future frontend tool needs approval, throw to force designer to reconsider.
      if (options.requiresApproval) {
        throw new Error(`Frontend tool '${options.toolId}' cannot require approval — move to agent.`);
      }

      // 3. Execute (no audit POST — see §9.1)
      fn(args);
    },
    [fn, options, grantedCapabilities],
  );
}
```

## Part 11 — Invariants

1. **No tool body executes without passing PolicyEngine.evaluate()** — the decorator is the only entry point; direct calls to wrapped functions bypass the decorator and are forbidden. Enforce via the startup registry assertion.

2. **Every tool in CREW_TOOLS must be registered in _TOOL_REGISTRY** — enforced at agent startup by `assert_all_tools_registered`. Missing registration is a fatal boot error, not a runtime warning.

3. **Future capabilities (db.*, fs.*, shell.*) are permanently denied in 3.3** — `_has_blocked_capability` hardcodes this. No role override can unblock them; the check is pre-role.

4. **Denied tools never execute** — `PolicyDecision.Deny` returns before calling `fn(*args, **kwargs)`. This must not be relaxed for any reason.

5. **Pending-approval tools never execute** — `PolicyDecision.PendingApproval` returns a sentinel dict without calling `fn`. Execution resumes only when the agent re-invokes the tool with a valid `_approval_token`.

6. **Audit log writes are non-blocking and never raise** — `AuditLogger.log` must catch all exceptions internally. A logging failure must not prevent tool execution from succeeding.

7. **SurfaceIntent literal is the canonical discriminant** — frontend routes exclusively on `envelope.intent`. No other field is used for routing. New intents require a corresponding handler before the literal is added.

8. **Frontend capability checks are UX guards only** — they cannot substitute for backend policy enforcement. Any security assumption must live in PolicyEngine, not in `useGuardedFrontendTool`.

9. **TS capability enum must match Python Capability enum exactly** — any drift causes silent mismatch between frontend guard and backend policy. Until a generator exists, both files must be updated in the same commit.

10. **ApprovalGate must expire** — `expires_at` is enforced by the BFF: after expiry, approve requests return 410 Gone. The frontend auto-dismisses on expiry (poll or server-sent event — implement in 3.4).

11. **RiskLevel in the tool declaration is immutable post-registration** — `ToolMeta` is a frozen dataclass. Risk levels are not runtime-configurable; they are code decisions reviewed in PR.

12. **The audit stub (console log) must be a drop-in replacement for the real DB writer** — `AuditLogger` must have the exact same call signature in 3.3 and 3.4. No call sites change when swapping the implementation.
